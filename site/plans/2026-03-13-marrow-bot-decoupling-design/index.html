marrow-bot Decoupling Design¶

Goal¶

Decouple all non-core bot behavior from marrow-core into a new marrow-bot repository so that:

  • marrow-core remains a minimal, reusable scheduler + runtime library.
  • marrow-bot becomes the canonical default bot profile (roles, prompts, context, config).

Target outcome matches Issue #32: move non-core mechanisms to marrow-bot while keeping the core service model and contracts stable.

Related: marrow 分层架构与 OKR/产出设计 — four-layer model (core / infra-plugin / application / evolution), OKR–Thread–Task relationship, outcome types per responsibility, and core extension for custom background services (e.g. embedded web / dashboard).

Core / Bot Boundary¶

What stays in marrow-core¶

  • Python packages
  • marrow_core/contracts.py
    • Defines workspace topology and configuration contracts.
    • Expresses how to locate roles, prompts, context, and runtime directories.
    • No hard-coded knowledge of specific role names or bot behaviors.
  • marrow_core/prompting.py
    • Assembles prompts from configured roles/, prompts/, and context.d/.
    • Treats these as external inputs, not as repo-owned content.
  • marrow_core/runtime.py
    • Socket, queue, and binary path resolution.
  • marrow_core/task_queue.py
    • Filesystem-backed queue operations.
  • marrow_core/services.py
    • launchd/systemd service rendering (marrow-heart.service, com.marrow.heart.plist).
  • marrow_core/scaffold.py
    • Workspace scaffold and starter config generation, parameterized by paths.
  • marrow_core/heartbeat.py
    • Scheduled orchestration per configured L1 main.
  • marrow_core/ipc.py
    • Local Unix-socket control plane.
  • marrow_core/cli.py
    • User-facing CLI surface (run, run-once, sync-once, setup, scaffold, validate, doctor, status, install-service, task *).

  • Any small helpers (config.py, sync.py, etc.) that are required to keep the above contract coherent.

  • Docs

  • AGENTS.md
    • Describes contracts: directory layout, IPC surface, CLI entrypoints, expected result codes.
    • Describes generic runtime layout under /opt/marrow-core and /Users/marrow.
    • Does not define concrete bot roles, behaviors, or handoff semantics.

  • README.md

    • Explains installation and upgrade of marrow-core as a library + service.
    • Documents CLI commands and configuration fields at a generic level.
    • Refers to external bot profiles (e.g. marrow-bot) for concrete roles and workflows.

  • Shell / service artifacts

  • setup.sh
    • Installs/updates the venv and marrow-core package.
    • Renders and installs the single heartbeat service for the platform.
    • Does not assume or manipulate a particular bot profile.
  • lib.sh, marrow-heart.service, com.marrow.heart.plist
    • Shared helper logic and templates required to run the heartbeat loop and sync model.

What moves to marrow-bot¶

  • Prompt & role content
  • roles/ tree (all l1/, l2/, l3/ roles).
  • roles.toml model-tier and casting configuration.
  • prompts/rules.md and any additional bot-specific rules.

  • Any future prompt/policy files that define concrete bot behavior or tone.

  • Context providers

  • context.d/*.py that express bot-specific monitoring, task generation, or workspace exploration.

  • Any additional context providers that are not strictly required for the core runtime to function.

  • Bot-specific configuration

  • Default marrow.toml that configures agents such as scout, conductor, and refit.

  • Recommended workspace paths and agent commands for the default bot profile.

  • Bot-facing docs and workflows

  • Detailed AGENTS.md for bot behavior:
    • Role responsibilities and delegation policy.
    • Handoff directory semantics (runtime/handoff/*).
    • Example workflows for human/agent collaboration.

  • A README.md in marrow-bot that explains:

    • How to install marrow-core.
    • How to install and enable the marrow-bot profile.
    • How to update both core and bot cleanly.

  • Bot installation helpers

  • Scripts such as setup-bot.sh that:
    • Validate that marrow-core is installed.
    • Cast roles/ into .opencode/agents/ via role-forge.
    • Copy or reference marrow-bot’s marrow.toml.
    • Optionally call marrow-core CLI (setup, install-service, sync-once) with the right config.

Design Principles¶

  • Core is profile-agnostic
  • marrow-core must not embed knowledge of specific roles (like scout, conductor, refit).
  • All such knowledge lives in marrow-bot (or any other bot profile).

  • Core APIs accept paths/configuration and operate on those, regardless of their contents.

  • Stable runtime contracts

  • Keep CLI surface, IPC protocol, directory layout, and exit codes stable across the split.

  • marrow-bot is responsible for staying within these contracts; marrow-core does not depend on bot internals.

  • Pluggable profiles

  • Treat marrow-bot as “just one profile” that consumes the marrow-core runtime.

  • Future profiles should be able to depend only on the documented contracts, not on private internals.

  • Minimal migration for existing users

  • Existing installations should be able to:
    • Upgrade marrow-core (runtime only).
    • Install marrow-bot to recover the current roles & behavior.
  • The service model (single heartbeat service + CLI-managed sync) must remain the same.

Migration Outline¶

  1. Abstract configuration and paths
  2. Audit marrow_core/contracts.py, runtime.py, scaffold.py, and related modules for assumptions that roles/, prompts/, or context.d/ live inside the marrow-core repo.

  3. Replace any hard-coded repo-relative paths with:

    • Configuration-driven paths from marrow.toml (core_dir, workspace, roles_dir, context_dirs, etc.).
    • Reasonable defaults that do not require repo-local prompt content.

  4. Move non-core content into marrow-bot

  5. Create a new marrow-bot repository.

  6. Copy the following from marrow-core into marrow-bot:

    • roles/
    • roles.toml
    • prompts/rules.md
    • context.d/*.py
    • The default marrow.toml (or equivalent), adjusted to reference marrow-bot locations as needed.

  7. Simplify core docs and setup

  8. Update README.md:
    • Focus on installing/updating marrow-core as a runtime.
    • Refer to marrow-bot for a default role/profile bundle.
  9. Update AGENTS.md:
    • Keep only architecture, filesystem layout, CLI/IPC contracts.
    • Remove role-specific behavior descriptions and move them to marrow-bot.

  10. Update setup.sh:

    • Remove any logic that assumes or manipulates repo-local roles/ or bot state.
    • Keep venv creation, package installation, and service installation.

  11. Introduce a clean bot installation path

  12. In marrow-bot, add:

    • A README.md documenting how to install marrow-core and then enable marrow-bot.
    • A helper script (setup-bot.sh or similar) that:
    • Ensures marrow-core is available.
    • Runs role-forge casting for roles/.
    • Copies or links marrow-bot’s marrow.toml into the workspace.
    • Invokes marrow-core CLI commands as needed to finish setup.

  13. Verify behavior parity

  14. Ensure that with both repos installed:
    • The effective runtime behavior (roles, handoffs, scheduling) matches the pre-split state.
  15. Ensure that with only marrow-core installed:
    • CLI and service installation still work.
    • Commands that require a profile fail clearly with actionable errors (“no roles/profile configured”).

Compatibility Considerations¶

  • Existing installations
  • Document a migration path:
    • Upgrade marrow-core in place.
    • Install marrow-bot and point marrow.toml to the new profile.

  • Consider adding a temporary compatibility note in README.md explaining the split and linking to marrow-bot.

  • Versioning

  • Bump marrow-core minor version to reflect the structural change.

  • Align marrow-bot’s initial version with the marrow-core version it expects.

  • Testing

  • Keep existing unit tests for marrow_core/* modules, adjusting imports and fixtures where they referred to repo-local roles/ or context.d/.
  • Add tests (in either repo) that:
    • Verify marrow-core behaves correctly with a minimal/no-profile config.
    • Verify marrow-bot integrates against the documented contracts.
