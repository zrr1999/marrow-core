marrow 分层架构与 OKR/产出设计¶

目标¶

在现有 marrow-bot 解耦设计 基础上，明确：

  1. 四层架构：核心层、基建/插件层、应用层、进化层；用户视角与 agent 职责边界。
  2. Core 扩展：支持自定义后台服务（含嵌入 Web），为后续 Dashboard 等插件提供基建；权限归属「谁嵌入谁拥有」。
  3. OKR ↔ Thread ↔ Task 关系，以及按职责的明确产出类型（PR、更新、试验结果、调研报告等；失败/证明不可行也算有效结果）。

本文档与解耦设计互补：解耦设计规定 core/bot 内容边界，本文规定层次模型、任务/产出语义与 core 的插件能力。

四层架构¶

层次定义¶

层次 名称 职责 用户视角 典型归属
核心层 Core 最基础能力：调度、队列、同步、服务安装、IPC、配置契约、prompt 组装与 context 执行 用户可见且依赖 marrow-core
基建/插件层 Infra / Plugin 可选的扩展能力：自定义后台服务、嵌入 Web（如 Dashboard）、插件注册与生命周期 用户可视为「也是核心」的一部分 marrow-core 提供机制；Dashboard 等以插件形式接入
应用层 Application 具体 bot 行为：角色、委托策略、任务管线、交互与自我工作流程 用户主要关心 marrow-bot（及未来其他 profile）
进化层 Evolution 自进化：造轮子、反思、指标收集、MetaTask、产出可优化清单 Agent 自主负责；用户不直接配置 marrow-bot 内 MetaTask + 进化流程

用户与 Agent 的视角¶

  • 用户：主要接触「核心层 + 基建/插件层」（可统称为「平台」）和「应用层」；不直接设计进化层策略。
  • Agent：在应用层执行用户任务与自我工作；在进化层自主执行反思、造轮子立项与实现、产出量化与报告。

高内聚原则：核心层不依赖应用层或进化层；应用层依赖核心层（及可选基建）；进化层是应用层内的一组专门流程与角色（如 evolver、scout），不反向依赖核心实现细节。

核心层扩展：自定义后台服务与嵌入 Web¶

目标¶

  • marrow-core 提供机制：允许「在 heartbeat 或独立进程中」注册并运行自定义后台服务。
  • 典型用途：嵌入一个 Web 服务（谁嵌入，权限与数据归属即归谁），后续可在此之上搭建 Dashboard，作为插件机制的核心内容。

设计要点¶

  1. 契约
  2. 在 marrow.toml（或等价配置）中声明可选的后台服务列表，例如：
    • 类型：web / worker / 其他预留。
    • 入口：命令行或模块路径（由 profile/插件提供）。
    • 绑定地址/端口、环境变量等由配置或约定决定。

  3. Core 只负责：根据配置启动/停止子进程、可选地做健康检查与重启策略；不关心服务内部业务逻辑。

  4. 权限与归属

  5. 「谁嵌入谁拥有」：若 marrow-bot（或某 profile）声明并嵌入一个 Web 服务，则该服务的访问控制与数据范围由该 profile 的部署者决定；core 不强制统一鉴权模型，只保证服务以配置的用户/环境运行。

  6. 与现有服务模型的关系

  7. 现有 heartbeat 仍为单一主调度服务（launchd/systemd 等）。
  8. 自定义后台服务可以是：
    • 与 heartbeat 同进程内多协程/多线程（若实现简单），或
    • 独立子进程由 core 的「服务管理器」拉起（推荐，隔离性好）。

  9. 实现时需避免与当前 render_service_files 冲突：可先支持「由 heartbeat 或单独 systemd/launchd 单元」拉起一个「插件宿主进程」，再由该宿主按配置启动 Web 等子服务。

  10. Dashboard 作为插件

  11. Dashboard 作为「基建/插件层」的典型应用：以可选插件形式提供，依赖 core 的「自定义后台服务」能力。
  12. 数据来源：通过 core 已暴露的 IPC/队列/状态接口或约定目录下的文件；不要求 core 实现 HTTP API，Dashboard 可读本地文件或 Unix socket。

具体实现可放在后续实现计划中（如扩展 marrow_core/services.py 与配置 schema）。

OKR、Thread、Task 关系与流程¶

概念层级¶

  • OKR（Objective / Key Result）
  • 顶层目标与关键结果，时间跨度较大（如一周、一月）。

  • 由应用层定义与维护；core 不解析 OKR 语义，只提供存储或透传（若需要）。

  • Thread

  • 一个连续工作流或主题线，对应一个 OKR 下的一个方向或一个「可独立执行的故事」。

  • 同一 Thread 内任务共享上下文（同一仓库、同一需求、同一实验）。

  • Task

  • 最小执行单元：有明确输入、执行步骤、必达产出类型（见下）。
  • 流程：任务要求 → 合并整理 → 加入/创建 Thread → 执行 → 产出记录。

流程（应用层，marrow-bot）¶

  1. 任务要求
  2. 外部输入（用户、Gateway、或 MetaTask）产生原始「任务要求」。
  3. 合并整理
  4. 规范化描述，映射到 OKR/Thread（若已有则挂到现有 Thread，否则创建 Thread，必要时关联或创建 OKR）。
  5. 加入/创建线程
  6. 确定或创建 thread_id，将任务作为 Task 入队，状态与优先级由应用层策略决定。
  7. 执行
  8. 由 L1/L2 角色按委托执行；core 提供队列与状态原语。
  9. 产出
  10. 每个 Task 必须落在一个明确产出类型上，并写入 Outcome（见下一节）。

OKR 与 Thread 的划分（供实现时细化）

  • 一个 OKR 对应一个周期内的顶层目标（如「本迭代交付 X」「建立 Y 能力」），可对应多个 Thread。
  • Thread 是 OKR 下的可执行工作流：同一需求、同一实验、同一仓库的连续任务归入同一 Thread；当主题切换或新需求/新实验开始时，可新建 Thread 并可选地挂到现有或新 OKR。
  • Task 永远属于某个 Thread；新建 Task 时若不存在合适 Thread，则先「合并整理」再创建 Thread 再入队。
  • Core 只关心 Task/Thread 的存储与调度；OKR 的创建、归档与展示可由应用层或 Dashboard 插件在 bot 侧维护。

按职责的产出类型（Outcome）¶

不同职责对应不同的必须产出。以下类型为应用层约定，core 仅提供通用的 Outcome 存储（如 task_id, outcome_type, success, payload, metrics），由 marrow-bot 赋予语义。

职责/角色类型 产出类型 说明 失败/否定结果
交付/仓库跟进（如 courier, git-ops） PR 代码或配置变更以 PR 形式落地 明确说明「未开 PR」原因 + 替代产出（如 issue、文档链接）
对外发布/广播（如 herald） 更新 发布说明、changelog、公告等可追溯的更新记录 说明未发布原因及后续计划
验证/实验（如 verifier） 试验结果 可复现的通过/失败、日志、指标 失败或证明不可行也算有效结果，需记录结论与复现路径
探索/调研（如 scout, researcher） 调研/研究报告 结构化结论、选项对比、建议 结论可为「暂无可行方案」并附依据
归档与记忆（如 archivist） 归档记录 索引、链接、摘要 记录「未归档」原因
自进化（如 evolver） 变更集 + 指标 实际改动（prompt/role/工作流/轮子）+ 效果指标或反思摘要 记录「未采纳」原因与替代方案

产出记录的通用字段（建议）¶

  • task_id, thread_id, outcome_type, success（布尔或三值：success / failure_with_result / blocked）
  • artifact_ref：PR URL、文件路径、报告路径等
  • conclusion：短文本结论（尤其对「失败但算结果」的情况）
  • metrics：可选键值对（耗时、次数等）

Core 只需提供「按 task 写入/读取 Outcome」的接口与存储格式（如 JSON/TOML 文件或 SQLite 表），应用层负责按上表约束填写 outcome_type 与必填字段。

与现有解耦设计的关系¶

  • 核心层 / 基建层 对应解耦设计中的「marrow-core 保留内容」；本文新增 core 的「自定义后台服务 + 嵌入 Web」能力，不改变「core 不包含具体角色与 bot 行为」的边界。
  • 应用层 对应 marrow-bot：角色、prompt、context、任务管线与产出语义均在 marrow-bot。
  • 进化层 是应用层内由 evolver/scout 等驱动的 MetaTask 与造轮子流程，产出类型中已包含「自进化」一行。

实现时建议顺序：

  1. 在 core 中落成任务/线程/Outcome 的存储契约与最小 API（若尚未完全具备）。
  2. 在 marrow-bot 中落成 OKR/Thread/Task 与按职责的产出类型规范，并写入 AGENTS.md。
  3. 再在 core 中增加自定义后台服务与嵌入 Web 的配置与启动机制，最后接 Dashboard 插件。
