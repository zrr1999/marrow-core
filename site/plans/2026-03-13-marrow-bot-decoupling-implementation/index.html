marrow-bot Decoupling Implementation Plan¶

For Claude: REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

Goal: Move all non-core bot behavior (roles, prompts, context, default config, bot-specific docs) out of marrow-core into a new marrow-bot repository, leaving marrow-core as a profile-agnostic runtime and scheduler.

Architecture: marrow-core exposes stable CLI, IPC, and filesystem contracts and knows nothing about concrete roles or bot workflows. marrow-bot (and future profiles) consume these contracts by providing roles/, roles.toml, prompts/, context.d/, and a marrow.toml configuration, plus helper scripts to integrate with the core runtime.

Tech Stack: Python (marrow-core), role-forge / OpenCode casting, shell scripts for setup, systemd / launchd for services, standard filesystem-based queues and state.

Task 1: Audit core assumptions about repo-local roles/prompts/context¶

Files: - Modify: marrow_core/contracts.py - Modify: marrow_core/runtime.py - Modify: marrow_core/scaffold.py - Modify: marrow_core/config.py (if present and relevant) - Modify: marrow_core/prompting.py - Test: tests/test_workspace.py - Test: tests/test_runtime.py

Step 1: Identify hard-coded paths and assumptions

Inspect the listed modules for: - Direct references to roles/, prompts/, or context.d/ under the repo root. - Assumptions that these directories always exist or must be shipped with marrow-core. - Implicit coupling between specific role names (e.g. scout, conductor, refit) and the runtime.

Step 2: Document current behavior

For each identified assumption, write a short inline note in the implementation plan (not code comments) describing: - What the assumption is. - How it should change to become profile-agnostic (e.g. use config-driven paths, allow absence of content).

Step 3: Add or update tests that express expectations

In tests/test_workspace.py and tests/test_runtime.py: - Ensure there are tests that can be adapted to cover: - Behavior when no roles/config/profile is present (clean error messages). - Behavior when roles/config/profile are provided via configuration paths.

Run:

cd marrow-core
uv run pytest tests/test_workspace.py tests/test_runtime.py -q

Expected: tests pass (or fail only on cases we plan to update in later tasks).

Task 2: Make core profile-agnostic in code paths¶

Files: - Modify: marrow_core/contracts.py - Modify: marrow_core/runtime.py - Modify: marrow_core/scaffold.py - Modify: marrow_core/prompting.py - Modify: marrow_core/config.py (if used for path defaults) - Test: tests/test_scaffold.py - Test: tests/test_services.py

Step 1: Parameterize roles/prompts/context paths

In the listed modules: - Replace any use of repo-relative roles/, prompts/, context.d/ with: - Paths derived from marrow.toml configuration (e.g. core_dir, workspace, roles_dir, context_dirs). - Reasonable defaults that do not require the repo to contain these directories. - Ensure that: - The system continues to support the current filesystem layout under /opt/marrow-core and /Users/marrow. - But those locations are not tied to a specific content bundle.

Step 2: Remove direct role name dependencies from core

In contracts.py, runtime.py, heartbeat.py, and related modules: - Remove or generalize any code that directly depends on role names like scout, conductor, or refit. - If role names are needed for tests or examples, move them to the test fixtures or docstrings, not core logic.

Step 3: Update scaffold behavior

In marrow_core/scaffold.py: - Ensure that workspace scaffolding: - Does not assume that canonical roles/ are present in the same repo. - Focuses on creating core runtime directories (runtime/state, runtime/handoff/*, etc.). - Leaves role/profile setup to external profiles (e.g. marrow-bot).

Step 4: Adjust and run tests

Update tests/test_scaffold.py, tests/test_services.py, and any related tests to: - Reflect the new profile-agnostic behavior. - Avoid depending on repo-local prompt/role content.

Run:

cd marrow-core
uv run pytest tests/test_scaffold.py tests/test_services.py -q

Expected: tests pass after adjustments.

Task 3: Extract non-core content into a new marrow-bot repository¶

Files (source in marrow-core): - Move: roles/ (entire tree) - Move: roles.toml - Move: prompts/rules.md - Move: context.d/*.py - Move: default marrow.toml (if present in the repo root and representing the current bot profile)

Files (target in new repo): - Create: marrow-bot/roles/ - Create: marrow-bot/roles.toml - Create: marrow-bot/prompts/rules.md - Create: marrow-bot/context.d/ - Create: marrow-bot/marrow.toml

Step 1: Create marrow-bot repository structure

In a new directory (outside marrow-core): - Initialize a new git repository marrow-bot. - Create a minimal pyproject.toml or equivalent to document dependencies on marrow-core and role-forge.

Step 2: Copy non-core content

From marrow-core into marrow-bot: - Copy roles/ into marrow-bot/roles/. - Copy roles.toml into marrow-bot/roles.toml. - Copy prompts/rules.md into marrow-bot/prompts/rules.md. - Copy context.d/*.py into marrow-bot/context.d/. - Copy the default marrow.toml into marrow-bot/marrow.toml, adjusting paths if needed.

Step 3: Verify imports and references

In marrow-bot: - Ensure any Python modules under context.d/ still import only public APIs from marrow-core. - Remove or fix any imports that referenced marrow-core internals that may change.

No tests to run yet unless we add them in a later task; just ensure the content is syntactically valid.

Task 5: Adjust setup.sh and related shell helpers¶

Files: - Modify: setup.sh - Modify: lib.sh (if it contains profile-specific logic) - Modify: any shell scripts under the repo root that reference roles/ or context.d/ - Test: run setup script in a controlled environment (e.g. local VM or container)

Step 1: Remove profile-specific steps from setup.sh

In setup.sh: - Remove logic that: - Assumes roles/ live inside the marrow-core repo. - Casts canonical roles into .opencode/agents/. - Manipulates bot-specific marrow.toml or profile configuration. - Keep logic that: - Creates/updates /opt/marrow-core/.venv. - Installs/updates the marrow-core Python package in that venv. - Renders and installs the single heartbeat service via marrow install-service.

Step 2: Confirm cross-platform behavior

Ensure that: - macOS path (com.marrow.heart.plist) is still rendered and installed correctly. - Linux path (marrow-heart.service) is still rendered and installed correctly.

Step 3: Dry-run setup

Run (without sudo if possible in a dev environment, or with a safe prefix):

cd /opt/marrow-core  # or the local checkout path
./setup.sh --help || true

If a dry-run or help mode is available, use it to verify that: - The script no longer refers to roles/ or context.d/. - The script focuses only on core installation tasks.

Task 6: Add marrow-bot installation helpers and docs¶

Files (in marrow-bot repo): - Create: README.md - Create: AGENTS.md (bot-focused) - Create: setup-bot.sh

Step 1: Write marrow-bot README

In marrow-bot/README.md, document: - Prerequisite: marrow-core installation. - How to: - Clone marrow-bot. - Run setup-bot.sh (or equivalent) to: - Cast roles via role-forge. - Place marrow.toml in the right location. - Optionally invoke marrow-core CLI to validate and install services.

Step 2: Write AGENTS.md for bot

In marrow-bot/AGENTS.md: - Describe: - Each role’s responsibility (scout, conductor, refit, etc.). - Delegation rules and handoff semantics. - Example flows between agents and human operators.

Step 3: Implement setup-bot.sh

Create a shell script that: - Verifies marrow-core is installed (e.g. checks for marrow CLI or core directory). - Runs role-forge (or agent-caster) to cast roles/ into .opencode/agents/. - Places or symlinks marrow-bot/marrow.toml to the intended config path (e.g. /opt/marrow-core/marrow.toml or a user-specified location). - Optionally invokes:

marrow validate --config /path/to/marrow.toml
marrow install-service --config /path/to/marrow.toml --platform auto --output-dir ./service-out

Ensure the script is idempotent and safe to re-run.
