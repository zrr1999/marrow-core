CLI-Integrated Sync Design¶

Goal¶

Replace the separate sync service and timer with a single cross-platform runtime model that keeps update behavior simple, bounded, and recoverable.

Constraints¶

  • Prefer one cross-platform implementation over platform-specific watchers or schedulers.
  • Keep the service model minimal: one long-running service, one sync path, one place to observe failures.
  • Avoid hot-reload complexity when a full process restart is simpler and safer.
  • Prevent sync failures from taking down the main heartbeat loop.

Current State¶

Today the project has two execution paths:

  • marrow run provides the main heartbeat loop.
  • sync.sh provides repository update, dependency refresh, workspace sync, and service reinstall.

This split creates a few problems:

  • Linux sync freshness depends on marrow-heart-sync.timer, currently only once per day.
  • macOS and Linux both carry a second service path just for sync.
  • The core update path lives in shell scripts, while the runtime logic lives in Python CLI code.
  • Sync and heartbeat failures are hard to reason about together because they happen in different processes with different entrypoints.

Proposed Model¶

Use one persistent service on both platforms:

  • macOS: com.marrow.heart.plist
  • Linux: marrow-heart.service

Move sync orchestration into the CLI with two layers:

  1. marrow run Runs the heartbeat loop and a lightweight periodic sync supervisor.
  2. marrow sync-once Performs exactly one sync attempt and returns a structured outcome.

The supervisor remains intentionally small. It only decides when to invoke sync-once, records basic state, and reacts to the result. It does not directly perform git, uv, or file mutation steps inside the main loop.

Why This Is The Minimal Cross-Platform Design¶

This approach keeps the implementation restrained:

  • No platform-specific file watching such as inotify or FSEvents
  • No separate sync service or timer lifecycle
  • No complex in-process hot-swapping of code or dependencies
  • No persistent worker pool or supervision tree beyond the main service manager

It uses the same mechanism on macOS and Linux: a normal long-running process with internal periodic polling.

Sync Execution Boundary¶

marrow run should invoke marrow sync-once in a subprocess, not by directly calling the sync logic inline.

Reasons:

  • git and uv are higher-risk operations than normal heartbeat scheduling
  • subprocess boundaries limit crash radius
  • return codes provide a simple contract
  • the main loop can keep running if sync fails transiently

This keeps policy in the runtime and dangerous mutation in a smaller execution unit.

Result Contract¶

marrow sync-once should return one of a very small number of outcomes:

  • noop: nothing changed
  • reloaded: only lightweight runtime data changed, and the current process can safely reload
  • restart_required: the current process should exit so the service manager restarts it
  • failed: sync attempt did not complete successfully

This can be represented by exit codes plus a small JSON state file under runtime/state/.

Suggested state fields:

{
  "last_attempt_at": "2026-03-08T00:00:00Z",
  "last_success_at": "2026-03-08T00:00:00Z",
  "result": "noop",
  "reason": "remote unchanged"
}

The state file is diagnostic only. The control flow should stay small enough that exit codes remain the primary contract.

What sync-once Does¶

marrow sync-once should absorb the useful parts of sync.sh:

  1. Check whether the core repository has a remote update available.
  2. Abort if the worktree is dirty.
  3. Fast-forward to the new revision when safe.
  4. Refresh dependencies when needed.
  5. Refresh workspace directories and role symlinks.
  6. Re-render service files if templates changed.
  7. Decide whether the live process can reload or should restart.

The command should stay conservative. If there is uncertainty about process safety after a change, prefer restart_required over a deeper hot-reload path.

Reload Rules¶

Safe to reload in-process:

  • role markdown changes
  • workspace directory creation
  • symlink refresh
  • config changes that only affect future scheduling state and can be reapplied cleanly

Require full restart:

  • CLI code changes
  • Python dependency changes
  • service template changes
  • changes that alter process bootstrap assumptions
  • any update whose safety cannot be cheaply proven

This rule set favors clarity over maximum liveness.

Crash Containment¶

The system needs only a few protections:

  • subprocess isolation for sync execution
  • a lock file so only one sync attempt runs at a time
  • exponential backoff after failures
  • dirty-worktree refusal to avoid destructive updates

These protections are intentionally enough, but not more than enough.

Service Rendering Changes¶

install-service should render only the primary service files:

  • com.marrow.heart.plist
  • marrow-heart.service

The repo should stop generating:

  • com.marrow.heart.sync.plist
  • marrow-heart-sync.service
  • marrow-heart-sync.timer

Related shell helpers should also be simplified so setup installs one service definition instead of maintaining a second sync lifecycle.

Testing Strategy¶

The main regression coverage should focus on:

  • service rendering no longer includes sync timer/service artifacts
  • install-service reports the reduced file set
  • sync-once outcome classification
  • run behavior when sync reports noop, reloaded, restart_required, or failure
  • failure backoff and lock behavior

The tests should prefer small unit and black-box CLI coverage over integration-heavy service tests.

Non-Goals¶

This design does not attempt to provide:

  • sub-second local file watching
  • zero-downtime code reloads after dependency or runtime changes
  • platform-specific optimizations
  • a separate supervisor daemon

Those can be added later only if the simple polling model proves insufficient.
